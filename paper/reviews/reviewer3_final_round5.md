## 致命点

### 1. 抽出同値仮定の未証明主張（§4.3 vs §6.2）
本文§4.3は「`proj_i(x)=some(y) ↔ E(x,y)` の強仮定を分解し、現実抽出器への接続を段階化した」と述べ、定理4.3（`preimage_eq_semanticPullback`等）で soundness/completeness 分離を機械検証している。

しかし§6.2実OSSケーススタディでは：
- 「regex 抽出層は本稿で証明対象ではなく、機械検証済みモデルの前段入力生成層として扱う」
- 「regex 抽出そのものの soundness/completeness を証明していない」
- 「定理4.3を本デモへ直接適用したとは主張しない」

と述べながら、一方で：
- 「RQ5: 抽出関係適合を `↔` だけでなく one-sided（sound / complete）にも分解できるか」→「§4.3（sound-only / complete-only / equality）で段階的に解決」

と結論している。

**矛盾**: RQ5の主張（抽出適合分解の解決）と§6.2の明示限界（regex抽出は未証明）が整合していない。定理4.3は `E` の抽象存在を証明したが、実抽出器への適用可能性を示していない。したがってRQ5は「理論上の分解可能性」のみ示しており、「現実抽出器への適用解決」とは言えない。

**最小修正案**: §10結論の「RQ5: §4.3で段階的に解決」を「RQ5: §4.3で理論上の分解を形式化。実抽出器への適用はRegex層の意味保存証明が必要（今後の課題、§9-限界4）」に修正。

### 2. 型対応の曖昧性（`D(i)` vs `carrier i`）
§2.1記法表は「`D_i` → `D i : SpecSet (carrier i)`」と述べ、`D(i)` を層 `i` の対象領域として導入している。しかし：

**本文§2.1での主張**:
- 「`D(i)` を分離する理由」として「仕様が語る対象範囲」と「許容される条件」の分離を挙げる
- 「`A(i) ⊆ D(i)` を明示することで、root側証人がdomain内に落ちること（RQ2）を定理として追跡できる」

**Lean実装 (`Definitions/Model.lean`)** を確認すると:
```lean
structure Layer (β : Type w) where
  D : SpecSet β
  A : SpecSet β  -- 本文では Ui i に対応
  admissible_subset_domain : A ⊆ D
```

ここで `carrier i` は型 `Type w` であり、`D i` はその上の述語 `SpecSet (carrier i) = carrier i → Prop` である。

**型レベルの混同**:
- 「対象範囲」を `D(i)` で表現するなら、`carrier i` 全体が対象型空間であり、`D(i)` はその中の「有効な値の条件」である。
- しかし本文は「`D(i)` = 対象領域」と述べ、あたかも `D(i)` が型レベルの制限であるかのように記述している（「root側証人がdomain内に落ちる」という表現は型制約を示唆）。

実際には `carrier i` が型レベル対象全体であり、`D(i) : SpecSet (carrier i)` は実行時の値レベル制約である。本文は型レベル（`carrier i`）と値レベル述語（`D i`, `A i`）の区別を明示せず、「領域」という語で混同している。

**最小修正案**: §2.1の `D(i)` 説明を「`D(i)` は層 `i` の搬送型 `carrier i` 上の対象領域述語（値レベル制約）であり、`A(i)` はその中で許容される条件述語。`carrier i` 自体は型レベルの対象空間全体を表す」に修正。§2.2の逆像定義も「`y ∈ D(i)` は `y : carrier i` が述語 `D i` を満たすこと」と明記。

### 3. RQ6の再現可能性主張と抽出器正当性の分離不足
**RQ6**: 「抽出関係適合を `↔` だけでなく one-sided（sound / complete）にも分解できるか」（本文§1）

しかし本文中には2つの異なる文脈がある：
1. §4.3: 定理4.3群（`preimage_eq_semanticPullback` 等）で抽象的な `E : α → carrier i → Prop` に対する soundness/completeness 分解を証明
2. §6.2, §7.5: 実OSSアーティファクト抽出パイプラインの「技術的再実行可能性」（source-lock, snapshot SHA256検証）

**結論§10のRQ6評価**:
- 「RQ6: §6.2, §7.5 で source-lock 付き抽出パイプラインの技術的再実行可能性を確認（解決）。抽出正当性（soundness/completeness）は RQ6 の対象外であり、限界 §9 に記載。」

**問題点**: RQ6の文面「抽出関係適合を…分解できるか」は理論的問題設定（§4.3で解決済み）を指すが、§10の評価は「技術的再実行可能性」（§6.2）を指している。これらは別の側面であり、RQ6が何を問うているのか曖昧である。

さらに§1のRQ6の文面には「抽出関係適合」と書かれており、これは定理4.3の `E` に対する adequacy（`preimage = semanticPullback`）を指すと読める。しかし§10では「抽出正当性は対象外」としている。

**最小修正案**: 
- §1のRQ6を「RQ6-theory: 抽象的抽出関係 `E` に対して soundness/completeness を分離定式化できるか」と「RQ6-practice: 実アーティファクト抽出パイプラインの再実行可能性（source-lock検証）を確保できるか」に分離。
- §10で「RQ6-theory: §4.3で解決（定理4.3群）」「RQ6-practice: §6.2, §7.5で確認（ただし抽出器の意味保存証明は今後の課題）」と個別評価。

## 根拠（NG判定の理由）

1. **論理整合性**: RQ5とRQ6の「解決」主張が、本文中の明示限界（regex未証明、抽出正当性は対象外）と矛盾している。機械検証部分（定理4.3）は抽象モデル上の分解可能性のみ示し、実抽出器への適用可能性は未実証である。
2. **型整合性**: `carrier i`（型レベル）と `D(i)`（値レベル述語）の区別が曖昧であり、「領域」という語が型制約と値制約を混同させている。Lean定義と本文記述の対応関係が不明瞭。
3. **中核定理追跡性**: 定理4.3（adequacy分解）は Lean で追跡可能だが、実OSS抽出デモ（§6.2）への接続が「未証明」と明記されており、RQ5/RQ6の「解決」主張を裏付けていない。結論で「解決」と述べながら限界で「対象外」とする構造は論理矛盾である。
