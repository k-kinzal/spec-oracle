## 致命点

### 1. RQ6評価範囲の齟齬
RQ6は「**実OSSアーティファクト抽出パイプラインの技術的再実行可能性**を確保できるか」と定義されているが、§6.2で明示されている通り「regex抽出そのものの soundness/completeness を証明していない」ため、**抽出器自体の正当性は未検証**である。にもかかわらず§10結論で「RQ6: §6.2, §7.5 で source-lock 付き抽出パイプラインの技術的再実行可能性を確認（**解決**）」と記載されている。

- 問題: 「再実行可能性」と「抽出正当性」を混同している。RQ6が「抽出パイプラインの再実行可能性」を問うているなら解決と主張できるが、定義に「抽出の正当性保証」まで含意されている場合、未解決である。
- 最小修正案: §10結論のRQ6対応記述を以下に変更:
  ```
  RQ6: §6.2, §7.5 で source-lock 付き抽出パイプラインの技術的**再実行可能性**を確認（解決）。ただし抽出器自体の soundness/completeness（抽出正当性）は§9限界に記載の通り対象外であり、RQ6が抽出正当性を含む場合は部分解決。
  ```

### 2. 評価主張の過大解釈リスク（PoCと一般化の分離不足）
§6.2は「n=3のconvenience sample」「数値境界制約のみ」という限定を明記しているが、§1表1.1「評価軸」の「抽出パイプライン実行可能性デモ」行で「PoC, n=3」と記載しているだけで、**PoCの限界が評価軸そのものに明示されていない**。査読者が表1.1だけを見た場合、「実行可能性デモが完了=一般化された手法」と誤読される可能性がある。

- 最小修正案: §1表1.1の該当行を以下に変更:
  ```
  | 抽出パイプライン実行可能性デモ | source-lock 付き再実行（**PoC限定**: n=3, 数値制約のみ, 抽出正当性は対象外） | §6.2（3 OSS, source-lock 再実行） |
  ```

### 3. 再現手順の実行順序不明確
§7.5「追試手順（オンライン/オフライン）」でオフライン追試の手順は記載されているが、**初回実行者（オンライン実行）が `--offline-lock` を指定せずに実行する手順との関係**が不明確である。特に:
- `external_validation.py` を引数なしで実行した場合、`external_validation_sources.lock.json` が既に存在する場合の動作（上書き/エラー/スキップ）が未記載
- リポジトリに既に `snapshots/*` が同梱されている場合、初回実行者が意図せずオフライン再生になるのか、オンライン取得が優先されるのかが不明

- 最小修正案: §7.5の「追試手順」に以下を追加:
  ```
  0. 初回実行者（オンライン）: `python external_validation.py` を引数なしで実行。既存の `source_lock` / `snapshots` は無視され、新規にオンライン取得・SHA256計算・snapshot書き込みが行われる。
  ```
