# 査読報告（査読者2: 再現性・評価担当）

**Verdict: NG**

## 致命的問題点

### 1. 評価主張の過大（PoC範囲超え）
**箇所**: §6.2, §9限界, 結論

**問題**:
- 本文は3件のconvenience sampleを「実OSS抽出パイプラインデモ」と称し、「抽出から判定までのパイプライン実行可能性デモ」として解釈するよう明記している（§6.2末尾）。
- しかし結論§10で「実アーティファクト抽出をソースロック付きで再実行可能にし、理論入力へ接続するPoC基盤を示した」と書く。
- §6.3では「実アーティファクト抽出をモデル入力へ再実行可能に接続できること」を主張するが、**regex抽出層のsoundness/completenessは仮定**と明記しながら、「理論との対応（最小接続）」で「層ごとの許容集合A(i)を区間述語として具体化したものとみなす」と記す。
- これは循環している。regex抽出が§4.3の前提（adequacy仮定）を満たす保証がないまま、「接続できた」と主張する構造になっている。

**判定根拠**:
- PoCとして許容されるのは「抽出スクリプトが実行でき、結果JSONが出力され、再現可能性記録（source lock）が残る」までである。
- しかし「理論入力へ接続する」と書くなら、regex抽出が定理4.3の`E: Ω → β_i → Prop`を実装していることの検証が最低限必要。
- 現状は「抽出結果をそのままA(i)として扱う」という**仮定の連鎖**であり、PoC範囲を超えて「接続基盤を示した」と主張するのは過大。

### 2. RQ6の扱いに論理飛躍がある
**箇所**: §1 RQ6定義, §6.2評価, §10結論

**問題**:
- RQ6は「実OSSアーティファクト抽出パイプラインを再実行可能な形で運用できるか（実行可能性）」と定義。
- §10では「RQ6: §6.2, §7.5でsource-lock付き抽出パイプラインの再実行可能性を確認（**部分解決**）。抽出正当性（soundness/completeness）の評価は**未解決**」と書く。
- しかし§6.2本文では「理論との対応（最小接続）」節で、抽出結果を`A(i)`とみなし、交差判定を`U∧`非空性判定に対応させたと記述している。

**判定根拠**:
- 「部分解決」と書きながら、§6.2では理論接続を主張する記述がある。これは内部矛盾。
- RQ6を「実行可能性」に限定するなら、§6.2の「理論との対応」節は削除すべき。あるいはRQ6定義を「理論接続の実証可能性」に拡張するなら、adequacy検証を追加すべき。
- 現状は「実行可能性だけ確認した」と「理論接続した」の間を曖昧に往復しており、主張範囲が不明瞭。

### 3. 再現性記述の不完全（依存関係の暗黙性）
**箇所**: §7.2, external_validation.py冒頭コメント

**問題**:
- §7.2は「Python標準ライブラリのみ使用、追加pip依存なし」と記す。
- しかし`external_validation.py`はインターネット接続を要求し、外部URL（postgresql.org, zlib.net, sqlite.org, github.com）への到達性を暗黙前提としている。
- source lockにはSHA256が記録されているが、これらURL先のコンテンツが将来変更された場合、再実行結果は異なる。
- snapshot機能はあるが、snapshotから逆算してsource lockを検証する手順が§7には書かれていない。

**判定根拠**:
- 再現性を主張するなら、「初回実行者はネットワーク必須」「追試者はsnapshot利用でネットワーク不要」の区別を明記すべき。
- 現状§7.5は「出力する」とだけ書き、追試者がsource lockとsnapshotをどう使えば再現できるかの手順が欠落。
- §7.2の「Python標準ライブラリのみ」という記述は、ネットワーク依存を隠蔽しており、誤解を招く。

## 最小修正案

### 修正1: §6.2「理論との対応」節を削除または大幅縮小
- 現行の「層ごとの許容集合A(i)を区間述語として具体化したものとみなす」記述を削除。
- 代わりに「抽出結果を入力として、区間交差判定スクリプトを実行可能にした」とだけ記述。
- 理論接続を主張したいなら、adequacy検証（regex抽出がsound/completeであることの根拠）を追加する。根拠なしに「対応する」と書くのは不可。

### 修正2: RQ6と§10結論の整合
- RQ6定義を「実行可能性の確認のみ」に限定するなら、§6.2の理論接続記述を全削除。
- あるいはRQ6を「理論入力への接続可能性の実証（adequacy仮定下）」に拡張し、adequacy仮定を明記する。
- §10結論では「抽出パイプラインの実行可能性を確認した。ただし抽出正当性（定理4.3のadequacy前提）は仮定であり、検証していない」と明記。

### 修正3: §7.5に追試手順を追記
以下を追加:
```
追試手順:
1. 初回実行者: `python external_validation.py`を実行（ネットワーク接続必須）。
   source lock, snapshot, 結果JSONが生成される。
2. 追試者: snapshotディレクトリを受け取り、スクリプトを修正してsnapshotから読み込むモードで実行。
   source lockのSHA256と一致することを確認。
3. ネットワーク不要な再現性を得るには、snapshot読み込み機能の実装が必要（現在未実装）。
```

または、snapshot読み込み機能を実装し、その旨を§7.5に明記。

## 総評
本稿はLean形式化（§2-5）とケーススタディ（§6）の接続において、adequacy仮定を明示している点は評価できる。しかし§6.2で「理論との対応」を主張しながら、adequacy検証を欠いたまま「接続基盤を示した」と結論で述べるのは、PoC範囲を超えた過大主張である。RQ6の「実行可能性」定義と§6.2の「理論接続」記述が矛盾しており、主張範囲が不明瞭。再現性記述もネットワーク依存と追試手順が不明確。上記3点を修正し、adequacy仮定を前提とした最小接続PoC（理論証明済み定理への**入力生成層**として扱う）として再構成すれば、査読通過可能な水準に達する。
